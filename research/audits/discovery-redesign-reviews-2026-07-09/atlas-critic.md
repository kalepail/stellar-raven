VERDICT: reject the atlas-first proposal as stated. The repository evidence does not show a live retrieval-collapse problem large enough to justify making a 4k-9k-token atlas the primary orientation surface for every conversation. A smaller, generated, on-demand service/workflow map may be worth testing, but the current proposal confuses three different failures: lexical retrieval misses, agent planning/query behavior, and downstream answer/data-quality failures. The first category is mostly already mitigated; the second can be attacked more cheaply by rewriting the existing instructions and supporting multi-query discovery; the third will not be fixed by an atlas.

1. The premise is overstated: the current eval record does not show a broad vague-question retrieval failure.

   The issue #9 problem statement is plausible as a model-behavior concern: it says agents compress ambiguous/status questions into one short query and should instead reason over source hypotheses before issuing multiple catalog searches. But that is not the same as "search cannot find the right operation." The committed eval map says the extended real-user-phrasing lane is diagnostic and records "pass@5 120/122, zero-hit 0" after tiered gate-rescue, down from 65/122 zero-hit, with strict top-1 77/122 after op keywords (eval/EVALS.md:17-22). The durable routing README makes the same point for the tiered rescue baseline: extended zero-hit 65/122 -> 0 and pass@5 -> 120/122 (eval/README.md:407-414).

   The eval runner also proves the routing lane is a direct test of raw-question lexical search, not an agent prompt artifact: `runCase` calls `searchCatalog(catalog, { query: c.question, limit: 5 })` and records the top hits (eval/run-routing.mjs:161-176). That is the harshest version of search-first: no atlas, no agent decomposition, raw full question.

   Current local result files are not even as flattering under strict labels, but they still do not support "zero-hit discovery failure." In `eval/results/routing-2026-07-09T12-01-50-970Z.json`, measured from the JSON, legacy strict is 213/267/305 of 338; extended strict is 79/104/110 of 122; extended accept-either is 110/121/122; zero-hit is 0. The 12 strict extended top-5 misses all return plausible non-empty hit lists, usually Scout or Lumenloop hits for questions whose `expected_any` includes Scout. That is a label/authority boundary issue, not absence of discoverable capabilities.

   Agentic evals reinforce the distinction. The 2026-07-06 agentic run shows docs at 100/100, Scout up, and Lumenloop primary down because upstream `scout.searchProjects` descriptions name-dropped products and captured editorial Lumenloop-labeled questions; the README calls 6/7 missed Lumenloop cases a documented boundary issue, one genuine misroute, and says the same-day QA headline was aggregate-identical to the prior best (eval/agentic/README.md:149-175). The filed upstream finding quantifies the same failure: 8 of 16 Lumenloop-labeled agent runs were captured by `searchProjects`, and the gateway's explicit counter-note was already present but lost to upstream name-drops (improvements/stellar-light-scout/sls-015-searchprojects-description-editorial-capture.md:27-36, 50-64). An atlas may restate the boundary, but the existing description already did; that weakens the claim that another prose surface fixes it.

2. The current instructions already contain most of the proposed atlas behavior.

   `SEARCH_DESCRIPTION` already tells the model to use search as a capability router, read signatures/descriptions, compose several operations in one execute script, fan out with `Promise.all`, use exact ids, use filters, treat hits as composable building blocks, read skills by section, run runnable skills, call `codemode.describe`, and perform deeper discovery inside execute with `codemode.search`, `codemode.catalog`, and `codemode.spec` (src/mcp/tools.ts:158-178). `EXECUTE_DESCRIPTION` repeats the workbench model, including mid-script discovery, exact `describe`, skill-reading, broad-to-detail followups, raw-row filtering before projection, and compact returns (src/mcp/tools.ts:180-218). `SERVER_INSTRUCTIONS` already includes workflow, source-authority framing, skills-vs-docs guidance, Scout/Lumenloop trust caveats, envelope interpretation, artifact reads, and exact-match ids (src/mcp/tools.ts:221-230).

   Therefore, a structured atlas has to prove it adds behavior beyond a better 1k-token instructions rewrite. The obvious low-cost change is to replace the brittle `search` step "short intent phrase (2-6 words)" with the #9 guidance: for ambiguous/status/research/current questions, issue several source-category searches before execute. That exact weak sentence is at src/mcp/tools.ts:162-166. The proposal has not shown that a new top-level/preloaded surface is required to teach that.

   A `kind:"workflow"` catalog lane or a short service/workflow section in `SERVER_INSTRUCTIONS` would cover most of the claimed benefit without turning operation cards into always-on context. The catalog kind set currently has only `operation`, `skill`, and `skill-section` (src/catalog/types.ts:33-36), so a workflow lane would be an additive discovery feature that can be measured independently.

3. The token math is not credible unless the atlas is aggressively summarized, and then it becomes hand-authored judgment.

   The current catalog is not "about 54 ops" in the exposed service-op sense. Current measured `catalog/manifest.json` has 272 entries generated at 2026-07-09T11:50:42.414Z: 50 operations, 18 skills, and 204 skill sections; operations split Lumenloop 18, Scout 20, StellarDocs 12. This matches the design record that says the current exposed surface is 50 service ops plus 18 skill entries plus 204 sections (PLAN.md:121-134; ADR-0003 follow-up at research/decisions/0003-build-time-exposure-filtering.md:94-106).

   Measured from the manifest with the repo's own 4-chars/token estimate, the 50 operation descriptions alone are 46,110 chars, about 11,528 tokens. A minimal `id: description` operation-card dump is 47,548 chars, about 11,887 tokens. Adding blank `useWhen`, `notFor`, and `followups` headings raises that to 50,142 chars, about 12,536 tokens before any actual extra content. All catalog descriptions together are 79,094 chars, about 19,774 tokens; operation schemas add about 66,284 chars, about 16,571 tokens, which is why schemas must stay out.

   The repo already fought this problem at search-hit scale: search output stubs oversized output types because a single limit-10 page carrying `scout.searchProjects` was about 26KB / 6.5k tokens, often attached to an off-target hit (src/catalog/search.ts:220-235; ARCHITECTURE.md:165-173). The execute result boundary is also about 6k tokens by default (ARCHITECTURE.md:217-219). Pushing 4k-9k tokens into every session is not free; it is roughly one whole default execute result of budget spent before the user asks anything.

   The issue #10 comment admits the risk: service+workflow-only atlas 800-2k tokens, operation-only compact atlas 2.5k-6k, atlas with operations/followups/source authority 4k-9k, skills summaries 8k-18k. The current manifest measurement says the ≤6k claim only holds if someone rewrites every operation into much shorter routing cards. That is not "generated from existing descriptions" anymore; it is a new editorial compression layer.

4. Authoring and drift are the biggest design debt, not an implementation detail.

   The repo's catalog pipeline is deliberately deterministic: `build-catalog.mjs` reads inventories plus the skill mirror, sorts entries, derives `generatedAt` from the newest input snapshot rather than wall clock, and emits byte-stable JSON (scripts/build-catalog.mjs:1-15, 817-835). Operation descriptions are generated from upstream inventory/spec data plus narrow exact-match notes: Lumenloop uses `description`, `when_to_use`, `returns`, and optional notes (scripts/build-catalog.mjs:350-382); Scout rewrites/scrubs upstream descriptions and appends exact notes (scripts/build-catalog.mjs:451-487); StellarDocs uses an authored spec-as-data description and returns field (scripts/build-catalog.mjs:525-560).

   The existing hand-authored prose mechanism is intentionally small and guarded. `description-notes.mjs` says notes are exact-match catalog guidance, shared by manifest and super-spec so surfaces cannot drift, and even constrains wording to avoid scorer blast-radius side effects (scripts/description-notes.mjs:1-12). Orphaned Lumenloop notes break the build (scripts/build-catalog.mjs:394-404). Orphaned Scout notes and scrubs break the build (scripts/build-catalog.mjs:499-520). Emitted text is scanned so non-exposed operation names, excluded endpoints, or retired skills cannot leak (scripts/build-catalog.mjs:732-777; scripts/emitted-text-guard.mjs:1-20, 40-71). ADR-0003 is explicit: the manifest is the exposed surface, excluded knowledge belongs in build-script data and ADRs, not runtime entries (research/decisions/0003-build-time-exposure-filtering.md:34-42, 94-117).

   A full atlas with `bestFor`, `notFor`, `useWhen`, `returns`, `commonFollowups`, and authority rules for every operation either duplicates those guarded descriptions or creates a second prose surface that can rot. If generated mechanically from current descriptions, it inherits their length and ambiguity. If hand-written, it expands the exact prose-drift problem that `description-notes.mjs` is designed to contain. Daily live drift makes this worse: upstream operation descriptions change and currently feed search/scoring with fail-loud guards; hand-maintained atlas cards would need equivalent orphan, leak, and scorer-regression discipline.

5. "Agents will use the atlas" is unproven, and the demo surface is actively hostile to a large always-on atlas.

   The repo only asserts initialize-time `SERVER_INSTRUCTIONS` surface in clients "like Claude Desktop" and outlive per-tool descriptions (src/mcp/tools.ts:221-224). The actual server passes `SERVER_INSTRUCTIONS` via `new McpServer(..., { instructions })` (src/server.ts:77-80). The browser playground manually injects production `SERVER_INSTRUCTIONS` plus a demo preamble as the model system prompt (src/demo/prompt.ts:1-17). That is not evidence that all MCP clients reliably present long server instructions or that models will scan a long atlas before calling tools.

   The public demo has tight caps: max 5 steps, 2 searches, 2 executes, 8k execute code chars, 4k output tokens, 20 replayed messages, and 24k history chars (src/demo/budget.ts:16-48). It also disables broad in-script discovery in the demo runner (`codemodeDiscovery: false`, `codemodeDescribe: true`) and the demo prompt tells the model not to call `codemode.search`, `codemode.catalog`, or `codemode.spec` inside execute (src/demo/tools.ts:49-56; src/demo/prompt.ts:17). A mandatory atlas would consume scarce context in exactly the surface where the repo has already clipped hit descriptions and signatures because full search pages stalled reasoning models (src/demo/tools.ts:69-80).

   The agentic evidence also cuts against faith in more prose. The 2026-07-06 boundary failure happened despite an explicit gateway note in the emitted `searchProjects` description; the upstream product-name prose overpowered it (improvements/stellar-light-scout/sls-015-searchprojects-description-editorial-capture.md:33-36, 60-64). More prose is not automatically more control.

6. The downstream QA failures mostly are not atlas-fixable.

   The 2026-07-06 QA checkpoint was 20 correct / 9 partial / 1 wrong, aggregate-identical to the prior best; plan regrade showed 29/30 required-covered and onPlanRatio 1.00 (eval/qa/README.md:245-253). The review of wrong/partial rows overturned 2 judge artifacts and left one wrong that was an upstream data gap, not agent routing: `q-scf-current-round` faithfully quoted stale Scout `scfRound` metadata (eval/qa/README.md:254-263). The corresponding finding says the answering agent "did everything right" and still got a wrong outcome because the service's curated SCF round field was false (improvements/stellar-light-scout/sls-014-scfround-meta-stale-at-birth.md:29-33).

   Other partials are likewise source or synthesis defects: Stellar Docs' Algolia crawler strips code blocks, making install commands unretrievable from the docs index (improvements/stellar-docs/sd-006-crawler-strips-code-blocks.md:14-28, 51-60); Lumenloop RWA roundups contain unscoped milestone numbers that a consumer cannot reconcile (improvements/lumenloop/ll-010-roundup-rwa-milestones-unscoped.md:13-36, 56-66). An atlas can tell the agent "use docs for CLI install" or "treat Lumenloop as community-aggregated"; it cannot put stripped commands into the index or repair stale Scout metadata.

7. The cheaper alternative is obvious and should be tested first.

   The issue #9 recommendation is mostly a prompt/tool-contract change: do not compress the full user question into one magic keyword; for ambiguous/status/research questions, reason about likely source categories, issue multiple catalog searches, then execute fan-out. The current search tool already supports narrowing by `kind`, `service`, and `limit` (src/mcp/tools.ts:45-71); search result `truncated` and `nextSteps` already tell the caller to raise limit or narrow/retry before concluding absence (src/mcp/tools.ts:102-120, 168-178). The demo nextSteps already explicitly allows a second search when hits are mismatched or truncated (src/demo/tools.ts:238-248).

   If one-search compression is the actual failure, add either (a) stronger multi-query guidance in `SEARCH_DESCRIPTION`/`SERVER_INSTRUCTIONS`, (b) a batch `search` input accepting `queries: string[]` and returning grouped compact pages, or (c) server-side query decomposition over service charters. All three are smaller than a new mandatory atlas and directly target the failure in #9. The repo has precedent for measured, narrow retrieval fixes: the real-user alias lane shipped `QUERY_TOKEN_ALIASES` after a 213-case alias-register instrument showed top-1 +15 / top-3 +19 / top-5 +6 strict with byte-identical control and offline lanes (eval/README.md:615-648). That is the bar an atlas should meet.

What a minimal defensible version would look like:

- Do not make atlas the primary mandatory preload.
- Start with a generated, schema-free, service/workflow micro-atlas under 1k-2k tokens: service purposes, source authority rules, and 8-12 workflow archetypes. No per-operation `notFor` unless it is generated from existing guarded description notes.
- Keep exact operation details behind `search` and `codemode.describe`; do not duplicate full operation descriptions.
- Put the first version in `SERVER_INSTRUCTIONS` or an on-demand `capabilities({ level: "services" | "workflows" })` tool, not in every search result and not as a replacement for search.
- If operation cards are added, generate them from manifest fields and `description-notes.mjs`, run `assertNoNonExposedRefs`, and add orphan/drift guards equivalent to the existing note/scrub guards. Treat any hand-authored card text as build-script data with tests, not free prose.
- First change the existing `SEARCH_DESCRIPTION` wording away from "one 2-6 word phrase" toward "several source-category routing searches for vague/status/research questions." This is cheaper and directly aligned with issue #9.

What eval would prove or disprove it:

- A/B three conditions, not two: current baseline; improved instructions + multi-query guidance; atlas + same improved guidance. If atlas only beats the old baseline, it has not proven unique value.
- Retrieval metrics: `eval/run-routing.mjs` legacy gate must not regress beyond `eval/gates.json`; extended lane must report strict and accept-either top-1/top-3/top-5 plus zero-hit; real-user alias/control lanes from round 844 should be rerun because they are unsaturated and independent of the offline corpus (eval/README.md:631-643, 668-669).
- Agent-behavior metrics: rerun `eval/agentic/` on the 30-case sample plus a new focused vague/status set containing issue examples (`protocol Zipper`, `CAP-62`, `latest recommendation`, `did project ship feature Y`) and current strict extended top-5 misses such as `q-crp-export-tx-history-taxes`, `q-defi-sdex-offer-lifecycle`, `q-pc-sequence-numbers-ordering-replace`, and `q-ti-friendbot-ratelimit-alternatives`. Track queriesUsed, distinct source families searched, primary/any service, and whether the agent searches more than once when first hits are off-target.
- End-to-end metrics: rerun the same 30-case QA sample and the live-data lane. Score correct/partial/wrong, tool calls, total cost, latency, and execute truncation. The atlas must improve true agent failures, not judge artifacts or upstream data gaps.
- Classification required per miss: retrieval miss, agent behavior miss, or downstream synthesis/source failure. Only the first two count as atlas wins. If most remaining misses still map to sls-014-style stale data, sd-006-style missing indexed content, ll-010-style unscoped source facts, or isolated synthesis slips, atlas-first is disproven.
